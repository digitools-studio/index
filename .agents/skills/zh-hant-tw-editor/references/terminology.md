# zh-Hant-TW Terminology

## 優先使用

| 建議詞 | 避免詞 | 備註 |
| --- | --- | --- |
| 資料 | 數據 | 「數據」可用於統計語境，但一般文章優先用「資料」。 |
| 導入 | 落地、賦能 | 描述系統或流程開始使用。 |
| 流程 | 閉環 | 使用具體流程名稱。 |
| 關鍵工作 | 抓手 | 直接描述要做的事。 |
| 使用者 | 用戶 | 台灣產品文章優先用「使用者」；LINE 官方語境可用「用戶」。 |
| 客戶 | 客戶端 | 指合作對象或服務對象。 |
| 程式碼 | 代碼 | 技術文章優先用「程式碼」。 |
| 部署 | 發佈、上線 | 視語境使用；技術部署用「部署」。 |
| 網站 | 站點 | 台灣一般讀者優先用「網站」。 |
| 設定 | 配置 | 台灣產品與操作文件優先用「設定」。 |
| 權限控管 | 權限控制 | 兩者皆可，文章中保持一致。 |
| 驗證 | 校驗 | 台灣常用「驗證」。 |
| 整合 | 集成 | 台灣常用「整合」。 |
| 需求釐清 | 需求梳理 | 台灣常用「釐清」或「整理」。 |

## 禁用或高度避免

- 賦能
- 抓手
- 閉環
- 痛點抓取
- 數據化賦能
- 打造生態
- 顛覆式
- 革命性
- 無痛導入
- 保證成長

## 技術詞彙第一次出現

- AI Native：第一次出現需解釋為「從產品流程與使用情境開始就把 AI 能力納入設計，而不是在既有系統外加聊天功能」。
- MVP：第一次出現需解釋為「最小可行產品，用最小範圍驗證關鍵假設」。
- API：第一次出現需解釋為「系統之間交換資料與指令的介面」。
- JSON-LD：第一次出現需解釋為「讓搜尋引擎理解頁面內容的結構化資料格式」。
- RBAC：第一次出現需解釋為「依角色管理可看、可改、可執行內容的權限模型」。
